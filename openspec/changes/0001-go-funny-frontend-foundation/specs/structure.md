# Spec: 基礎架構與頁面骨架

## Requirement 1
專案應提供可導航的核心頁面骨架，讓使用者可以從首頁進入旅程清單與單一旅程流程。

### Acceptance Criteria
- 首頁存在且可作為產品入口
- 可預留旅程清單、旅程明細、旅程管理、新增記錄、結算、邀請接受等路由
- 路由名稱與頁面責任一致，不混用管理頁與明細頁職責

## Requirement 2
系統應先定義 GO-FUNNY 的領域名詞，讓後續規格可沿用一致語言。

### Acceptance Criteria
- 旅程、成員、一般支出、公積金存入、公積金支出、結算等名詞有一致定義
- 一般支出與公積金記錄被視為不同記錄類型
- 結算頁需能區分一般結算與公積金結算

## Requirement 3
頁面資訊架構需符合 mobile-first 與簡潔原則。

### Acceptance Criteria
- 新頁面預設以手機版瀏覽為主
- 明細頁只承載記帳與結算相關資訊
- 旅程管理功能獨立在管理頁，不混入明細頁
